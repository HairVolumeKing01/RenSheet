# 动漫周边拼团 xlsx 肾表解析通用指南

## 适用场景

解析拼团/排谷场景下的成本分摊表（肾表），从xlsx中提取每个参团成员的具体明细。xlsx格式由团长手工制作，无统一规范，本指南覆盖常见变体。

---

## 四种肾表格式

### 格式1：CN纵列型

**特征**：一行一个成员（CN），明细写在一个单元格内，肾在末尾列。

```
| CN   | 明细              | 点数 | 肾   |
|------|-------------------|------|------|
| 张三 | 吧唧-椿1, 卡-樱2 | 3    | 45.0 |
| 李四 | 文件夹-兔1        | 1    | 12.0 |
```

**解析**：正则拆分明细文本 →

```
(\S+?)-(\S+?)(\d+)
```

`品类-角色数量`，用逗号/换行分隔多条。

---

### 格式2：角色散点型

**特征**：第2行=角色名（列头），第3行=单价，第4行起每个格子如果填了CN名就是该成员在这个角色下吃了1点。

```
| 序号 | 椿   | 樱   | 兔   |
|------|------|------|------|
|      | 12.0 | 15.0 | 8.0  |  ← 单价行
| 1    | 张三 |      |      |
| 2    |      | 张三 | 李四 |
| 3    | 李四 | 李四 |      |
```

**解析流程**：
1. 读 Row2（角色名）和 Row3（单价）建立 `col_idx → {role, price}` 映射
2. 从 Row4 遍历到数据区末尾：每格若为合法CN名 → `cn_stats[cn][role].qty += 1`
3. 同一CN同一角色可能出现多次（如张三在第1行和第5行都填在「椿」列），必须**累加**，不能覆盖

**⚠️ 常见陷阱**：
- Row0可能是序号列，角色从col1开始
- 单价行可能含非数字文本（如「序号/单价」），`float()` 前需 `isdigit()` 或 `try/except`
- 同一sheet可能有左右两个区域（如左侧卡区col1-11，右侧吧唧区col12-22），需指定 `col_start/col_end` 分段解析

---

### 格式3：CN行 × 品类列

**特征**：CN在行（通常col0或col5），品类名在列头，交叉格=数量，肾在末尾列。

```
| CN   | 吧唧 | 卡 | 文件夹 | 肾   |
|------|------|----|--------|------|
| 张三 | 2    | 1  | 0      | 39.0 |
| 李四 | 1    | 0  | 1      | 20.0 |
```

**解析流程**：
1. 找表头行（含「吧唧」「卡」等品类关键词）
2. 找单价行（通常在表头下一行）
3. 遍历数据行：CN列取值，各品类列取数量（跳过0和空），肾取末尾列

**变体**：单元格可能填 `角色数量` 而非纯数字（如 `椿2,樱1`），需额外split。

---

### 格式4：排表型（面交排表）

**特征**：非标准结构，无通用解析方案。特征包括角色×CN矩阵（交叉格为分配数量）、多层表头、合并单元格。建议人工预处理或针对单个文件写特化解析。

---

## 通用解析规则

### 编码

- CSV文件来自WPS/金山文档导出 → **GBK编码**（不是UTF-8）
- xlsx内部是XML，无编码问题

### xlsx读取

优先 `openpyxl.load_workbook(path, data_only=True)`。

不可用时用标准库：
```python
import zipfile, xml.etree.ElementTree as ET
# 1. 解析 xl/sharedStrings.xml 获取共享字符串表
# 2. 遍历 xl/worksheets/sheet*.xml 读取单元格
# 3. c.get('t') == 's' → 值在共享字符串表；否则直接读 <v> 文本
```

### CN合法性校验

- 维护合法CN白名单（从排谷明细CSV的cn列去重获得）
- 精确匹配失败后，用**单字包含搜索**做模糊匹配（如「沈」匹配「沈三召」「沈三沼」），避免因一字之差报「未找到」

### 日期处理

- xlsx内日期存储为序列号：`datetime(1899, 12, 30) + timedelta(days=int(serial))`
- CSV中日期通常为 `2026/6/1` 格式（月日无前导零），解析用 `%Y/%m/%d`

### 文件名过滤

- `~$` 开头 → WPS临时文件，跳过
- 仅含「退补」「国际运费」等关键词的xlsx → 非肾表，跳过

## 写入数据模型

```json
{
  "kidney_detail": [
    {"role": "椿", "qty": 2, "price": 12.0, "cat": "吧唧"}
  ],
  "kidney_total": 24.0
}
```

- `kidney_total` = 纯本体肾（谷子本身价格），**不含国际运费**
- `role` 相同的条目应合并：`qty` 累加，`price` 取其一（同角色单价相同）
- `cat` 可选字段，用于区分同一肾表中不同品类

---

## 多文件/多Sheet合并规则

### 多Sheet ≠ 多条目

一个xlsx含多个sheet（如「明信片」sheet + 「吧唧」sheet），当它们对应CSV中**不同条目**时，必须分别写入各自的 `kidney_detail`，不能把所有sheet的明细合并后写入每个条目。

### 跨文件合并

一个CSV条目可能对应多个xlsx文件（如第一批.xlsx + 第二批.xlsx）。解析时必须扫描所有关联文件，合并明细后写入**同一个** `kidney_detail` 数组。退补金额从退补表扣除。

## 致命Bug清单

| 序号 | Bug | 原因 | 修复 |
|:--:|------|------|------|
| 1 | **同CN同角色多行未累加** | 散点型中同一CN在同一角色列出现多次，只取了第1次 | 遍历时 `qty += 1`，跨sheet也要累加 |
| 2 | **多sheet明细混入** | 将所有sheet的明细合并后写入每个item | 按sheet分别匹配对应CSV条目 |
| 3 | **跨文件漏合并** | 只解析了第一个关联文件 | 扫描目录下所有匹配文件后汇总 |
| 4 | **name空白字符不匹配** | xlsx的name含全角空格，CSV不含 | 匹配前做 `replace(' ', '').replace('\u3000', '')` |
| 5 | **CSV点数≠实际点数** | CSV「数量」列记录的是批数，肾表实际是批数×每批点数 | 以肾表明细为准，不砍点 |
| 6 | **sent字段非标准值** | XML直接读出「未发走」等文本 | 写入时规范化为 `"0"` / `"1"` |

## 批量缓存策略

每次查询时现读xlsx极低效。应：

1. 启动时批量解析所有肾表 → 写入JSON缓存
2. 后续查询直接读缓存（秒出）
3. 监听肾表文件 mtime 变化，触发增量刷新
4. 记录 `kidney_cache_time` 做版本判断
